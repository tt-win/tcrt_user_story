# Automation Hub — Security Model

## Encryption at rest

Provider credentials (GitHub PAT, GitHub App private key, Jenkins API token,
Allure bearer token, etc.) are encrypted with **AES-256-GCM** before being
written to `team_automation_providers.credentials_encrypted`. Each row uses a
fresh 96-bit nonce; the envelope stored is JSON:

```json
{
  "v": 1,
  "alg": "AES-256-GCM",
  "nonce": "<base64>",
  "ciphertext": "<base64>",
  "fingerprint": "api_token:***xxxx"
}
```

The encryption key is loaded by [provider_credential_service](../app/services/automation/provider_credential_service.py) from:

1. Environment variable `AUTOMATION_PROVIDER_ENCRYPTION_KEY` (preferred)
2. `automation_provider.encryption_key` in `config.yaml` (fallback)

The key MUST be a base64-encoded 32-byte value. Generate one with:

```bash
python3 -c "import os, base64; print(base64.b64encode(os.urandom(32)).decode())"
```

### Operational rules

- **Never rotate the key after data exists.** Rotating loses decryption ability
  for every stored credential. To rotate, you must re-enter every provider's
  credentials after switching keys.
- Bootstrap (`database_init.py`) **blocks startup** when
  `team_automation_providers` has rows but the key is missing or malformed.
- API list / detail responses never return the raw secret — only a fingerprint
  derived from the encrypted envelope. Create / regenerate-secret responses
  return the plaintext **once**; the UI surfaces a one-time reveal modal.

## Webhook HMAC

Both inbound and outbound webhooks use **HMAC-SHA256** over the raw request
body with the per-webhook `secret`. Headers:

- `X-TCRT-Signature: sha256=<hex>` (also accepts bare `<hex>`)
- `X-TCRT-Delivery: <unique-id>` (best-effort idempotency hint)

Constant-time compare via `hmac.compare_digest`. Signature mismatch yields
HTTP 401 with `WEBHOOK_SIGNATURE_INVALID`.

Webhook secrets are stored **plaintext** in `automation_webhooks.secret`
(`VARCHAR(128)`) by design: HMAC verification needs the raw key, and storing
encrypted would require decrypt-on-every-request with no real security gain
(the DB itself is the trust boundary). If your threat model demands envelope
encryption for the secret, migrate the column to `MediumText` and reuse the
provider AES helper — the change is local to `webhook_service._generate_secret`
and `verify_signature`.

## Permission boundaries

| Surface | Required permission |
|---|---|
| Create / update / delete providers, suites, scripts, links, webhooks | Team **ADMIN** |
| Trigger script / suite run, cancel run, reconcile | Team **ADMIN** |
| List / read scripts, runs, coverage, linked-automation | Team **READ** |
| Inbound CI webhook (`/api/v1/webhooks/ci/{token}/run-status`) | Token + HMAC, **no user auth** |
| MCP automation-* read endpoints | Machine token with `mcp_read` permission |

## Provider authentication recommendations

| Provider | Recommended | Rationale |
|---|---|---|
| GitHub | **GitHub App** | Scoped, expiring tokens; central revocation; per-installation auditability. |
| GitHub (single repo) | Fine-grained PAT | Acceptable for small teams; rotate every 90 days. |
| Jenkins | API token + username (Basic Auth) + CSRF crumb | Per-user revocation; CSRF crumb mitigates form-replay. |
| Jenkins (job-only) | `default_job_token` (no Basic Auth) | OK for trigger-only flows; can't read job status. |
| Allure | Optional `api_token` | Only if Allure is gated by auth. |
| LocalGit | SSH deploy key (`ssh_key_path`) | Air-gapped; key file outside repo, mode `0600`. |

## What TCRT never stores

- Test data values, screenshots, Playwright traces, video artefacts — those
  remain in CI / report storage.
- The script content as source of truth — `cached_content` is best-effort
  cache that can be regenerated by `Rescan`.
- CI runner credentials — TCRT only sees the CI dispatcher API token.

## Auditing

Every mutation writes to the audit log via `AuditService`:

- `AUTOMATION_PROVIDER` — CRUD + test-connection
- `AUTOMATION_SCRIPT` — auto-discovery, metadata update, cache delete
- `AUTOMATION_SCRIPT_LINK` — create / update / delete link
- `AUTOMATION_SCRIPT_GROUP` — suite CRUD (run trigger now lives in
  Test Run Set; see
  [move-automation-execution-to-test-run-set](../openspec/changes/move-automation-execution-to-test-run-set/))
- `AUTOMATION_RUN` — trigger / cancel / reconcile / webhook ingest
- `AUTOMATION_WEBHOOK` — CRUD + regenerate-secret + test ping

DELETE actions are tagged `AuditSeverity.CRITICAL`; everything else is `INFO`.
